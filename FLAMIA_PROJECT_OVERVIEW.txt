1. SYSTEM DESCRIPTION
================================================================================

Flamia is a direct-to-consumer (D2C) luxury scented candle e-commerce platform
designed to deliver a cinematic, immersive shopping experience. The platform
serves the Indian market with phone-based OTP authentication, Razorpay payment
integration, and a manual UPI fallback mechanism.

The system is built as two separate applications:
  - A Spring Boot 3 backend providing REST APIs, security, and business logic
  - A React (Vite) frontend delivering a luxury animated user interface

The platform targets premium customers who value aesthetic presentation,
storytelling, and curated product experiences. Every interaction — from landing
page scroll animations to product detail transitions — is designed to feel
cinematic and intentional.

Key Differentiators:
  - Cinematic scroll-driven storytelling on the landing page
  - Phone OTP-only authentication (no passwords)
  - Rich media reviews (text, images, audio — no video)
  - Manual UPI payment fallback with screenshot + UTR verification
  - Admin-managed shipping via Shiprocket with manual tracking ID entry

2. BUSINESS GOALS
================================================================================

2.1 PRIMARY GOALS

  BG-01  Launch a D2C luxury candle brand with an online-first sales channel.
  BG-02  Deliver a premium brand experience through cinematic web design.
  BG-03  Enable frictionless purchasing for Indian customers via phone OTP
         and Razorpay/UPI payment methods.
  BG-04  Build trust through verified purchase reviews with rich media.
  BG-05  Minimize operational costs by deploying on free-tier infrastructure
         (Render, Vercel, Supabase).

  BG-06  Maintain a coupon and discount system to drive repeat purchases.
  BG-07  Enable admin to manage the complete product catalog, orders, and
         reviews without direct database access.
  BG-08  Collect customer data (addresses, order history) for future
         marketing initiatives.
  BG-09  Support future scalability to a microservices architecture if
         business growth demands it.

3. USER ROLES & PERSONAS
================================================================================

3.1 GUEST (Unauthenticated User)

  Permissions:
    - Browse landing page
    - View product catalog
    - View product detail pages
    - View published reviews
    - Search and filter products
    - View "About Us" and policy pages

  Restrictions:
    - Cannot add items to a persistent cart
    - Cannot place orders
    - Cannot submit reviews
    - Cannot access account pages

3.2 CUSTOMER (Authenticated User)

  Authentication Method: Supabase Phone OTP (Indian mobile numbers, +91)

  Permissions:
    - All guest permissions
    - Add items to cart (persisted via Zustand + localStorage)
    - Place orders
    - Pay via Razorpay or manual UPI
    - View order history and order status
    - Submit reviews for purchased products (text + images + audio)
    - Edit/delete own reviews
    - Manage delivery addresses
    - View and apply coupon codes
    - Receive email notifications (order confirmation, shipping updates)

  Restrictions:
    - Cannot access admin panel
    - Cannot review products they have not purchased
    - Cannot upload video reviews
    - Cannot upload images > 3MB or audio > 3MB

3.3 ADMIN (Administrative User)

  Authentication Method: Supabase Phone OTP + admin role flag in database

  Permissions:
    - Full product CRUD (create, read, update, delete)
    - Upload product images to Supabase Storage
    - View all orders
    - Update order status (CONFIRMED, SHIPPED, DELIVERED, CANCELLED)
    - Add Shiprocket tracking IDs to orders
    - Verify manual UPI payments (approve/reject based on screenshot + UTR)
    - Create, update, deactivate coupon codes
    - Delete customer reviews (cannot edit them)
    - View platform analytics (order counts, revenue summaries)

  Restrictions:
    - Cannot edit customer reviews (only delete)
    - Cannot modify customer account details directly

5.2 ARCHITECTURE STYLE: MONOLITHIC LAYERED

The backend follows a strict layered architecture:

  Layer 1 — Controller Layer (REST API Endpoints)
    Handles HTTP request/response mapping, input validation,
    and delegates to the service layer.

  Layer 2 — Service Layer (Business Logic)
    Contains all business rules, orchestration logic,
    validation, and external service integration.

  Layer 3 — Repository Layer (Data Access)
    Spring Data JPA repositories for database operations.
    No business logic in this layer.

  Cross-Cutting Concerns:
    - Security (Spring Security + Supabase JWT validation)
    - Exception handling (Global exception handler)
    - Logging (SLF4J + Logback)
    - Configuration (application.yml, environment variables)

5.3 DATA FLOW

  Guest Browsing:
    Browser → Vercel CDN → React App → Axios → Spring Boot API → PostgreSQL

  Authentication:
    Browser → Supabase JS SDK → Supabase Auth (OTP) → JWT issued
    Browser → Axios + JWT Header → Spring Boot (JWT validation) → Authorized

  Order Placement:
    React → API (create order) → Spring Boot → PostgreSQL
    React → Razorpay SDK → Razorpay → Webhook → Spring Boot → Update Order

  Manual UPI Payment:
    React → Upload screenshot to Supabase Storage → API (submit UTR + URL)
    Admin → API (verify payment) → Update order status

  Review Submission:
    React → Upload media to Supabase Storage → API (submit review with URLs)
    Spring Boot → Validate purchase history → Save review

7. REPOSITORY STRUCTURE
================================================================================

Two separate Git repositories:

  Repository 1: flamia-backend
    - Spring Boot 3 application
    - Contains all backend source code, configurations, and migrations
    - Deployed to Render

  Repository 2: flamia-frontend
    - React (Vite) application
    - Contains all frontend source code, assets, and configurations
    - Deployed to Vercel

Rationale for separation:
  - Independent deployment cycles
  - Different hosting platforms (Render vs Vercel)
  - Different CI/CD pipelines
  - Clear ownership boundaries
  - Frontend can be rebuilt/redeployed without backend changes

9. NON-FUNCTIONAL REQUIREMENTS
================================================================================

9.1 PERFORMANCE

  NFR-01  Landing page initial load         < 3 seconds on 4G
  NFR-02  API response time (95th pct)      < 500ms
  NFR-03  Image optimization                WebP format, lazy loading
  NFR-04  Frontend bundle size              < 500KB gzipped
  NFR-05  Database query time (95th pct)    < 100ms

9.2 SECURITY

  NFR-06  All API endpoints require HTTPS
  NFR-07  JWT tokens validated on every authenticated request
  NFR-08  CORS restricted to frontend domain only
  NFR-09  File upload size limits enforced server-side
  NFR-10  SQL injection prevention via parameterized queries (JPA)
  NFR-11  XSS prevention via React's default escaping + CSP headers
  NFR-12  Rate limiting on OTP requests (Supabase built-in)

9.3 RELIABILITY

  NFR-13  Graceful degradation when Razorpay is unavailable (UPI fallback)
  NFR-14  Retry logic for email sending (3 attempts with exponential backoff)
  NFR-15  Transaction management for order placement + payment creation

9.4 SCALABILITY (FUTURE)

  NFR-16  Stateless backend (no server-side sessions) for horizontal scaling
  NFR-17  Database connection pooling via HikariCP
  NFR-18  CDN-served frontend (Vercel Edge Network)

9.5 MEDIA CONSTRAINTS

  NFR-19  Image uploads: max 3MB per file
  NFR-20  Audio uploads: max 3MB per file
  NFR-21  Video uploads: NOT permitted
  NFR-22  Accepted image formats: JPEG, PNG, WebP
  NFR-23  Accepted audio formats: MP3, WAV, M4A

10. GLOSSARY
================================================================================

  Term                Definition
  ──────────────────  ──────────────────────────────────────────────────────
  D2C                 Direct-to-Consumer; selling directly without
                      intermediary retailers
  OTP                 One-Time Password; a numeric code sent via SMS for
                      authentication
  JWT                 JSON Web Token; a compact token format used for
                      secure information exchange
  UTR                 Unique Transaction Reference; a reference number
                      generated by banks for UPI transactions
  UPI                 Unified Payments Interface; an instant real-time
                      payment system by NPCI (India)
  Razorpay            Indian payment gateway supporting cards, UPI,
                      netbanking, and wallets
  Shiprocket          Indian shipping aggregator for order fulfillment
  Supabase            Open-source Firebase alternative providing auth,
                      database, and storage
  LCP                 Largest Contentful Paint; a Core Web Vital metric
                      measuring load performance
  HikariCP            High-performance JDBC connection pool used by
                      Spring Boot
  CSP                 Content Security Policy; HTTP header to prevent XSS
  CORS                Cross-Origin Resource Sharing; browser security
                      mechanism for cross-domain requests
